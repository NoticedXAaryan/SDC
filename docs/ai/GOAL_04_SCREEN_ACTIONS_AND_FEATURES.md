# Goal Pack 04 — screen actions, contextual controls, and feature completion map

## Purpose

This is document 4 of 5. It defines exactly what users must be able to do on each important screen. Implement these actions with the shadcn/Lucide system in Goal Pack 03 and with the authoritative backend model from `GEMINI_FLAGSHIP_PLATFORM_BLUEPRINT.md`.

The response to poor workflow is not “more screens.” It is a reliable sequence of meaningful controls at the moment and place the user needs them.

## Global screens

### Dashboard

Route family: `app/(dashboard)/dashboard/**`.

Required controls:

- Global command trigger, notification action center, profile menu, and role-aware Create menu in the shell.
- A single role-specific primary CTA:
  - Member: “Explore events”.
  - Lead: “Create event”.
  - Admin: “Review approvals” when pending approvals exist, otherwise “Create event”.
- Action queue cards with direct buttons: `Review`, `Complete registration`, `Open pass`, `Scan now`, `Retry delivery`, `Resolve import`, `View details`.
- Date-filtered “Today and next 7 days” event list with `Open scanner` only for authorized staff.
- Quick action card/grid, constrained to user permissions and actual existing workflows.
- “View all” links that preserve relevant scope in query parameters.

Feature requirements:

- Deadline/expiry awareness: registration closes, invite expires, certificate/campaign/import retry needs attention, interview occurs soon.
- Action queue is generated by real server state, not mocked counts.
- Dismissible advisory items must be user-scoped and retained; critical compliance/security items cannot be silently dismissed.
- Dashboard metrics always link to a filtered operational list.

### Work queue / approvals

Route: canonical `/approvals` or `/manage/approvals`; create redirects from duplicate destinations.

Required layout:

- `PageHeader` with title, description, count Badge, and filters.
- Data Table or compact cards with type, resource, requester, submitted time, current state, assigned reviewer, SLA/age, and actions.
- Filter chips: type, state, domain, assignee, age; URL backed.
- Row `Review` opens a Sheet with resource summary, relevant details, audit timeline, comments/reason input, and exact decision buttons.
- `Approve`, `Request changes`, and `Reject` are not icon-only. Rejection/request changes require a reason; consequence text appears before confirmation.
- Bulk decisions are disabled by default and allowed only for explicitly safe homogeneous requests; every decision is audited individually.

## Events: the highest-value operational module

### Events list

Routes: `/events` for all authorized users; role/permission determines controls, not separate duplicated list pages.

Required controls:

- Header primary action `Create event` for authorized roles; secondary `Calendar view`/`List view` toggle only if both views are implemented.
- Search by title/slug/location; filters for date, status, visibility, type, domain, registration state; `Clear filters` action.
- Event cards/table show cover, status, schedule/timezone, location, capacity/registration count, relevant deadline, owner/domain, and one context-aware CTA.
- Member actions: `View event`, `Register`, `View registration`, `Open pass`, `Join waitlist`.
- Lead/admin actions: `Manage`, `Edit`, `Duplicate`, `Open scanner`, `Export roster`, `Send update`, `Archive` in appropriate hierarchy.
- Empty states distinguish “no events exist” from “filters have no matches”.

Do not show both a giant “Register” button and a row-menu Register action. Choose one clear action based on list density/context.

### Event creation and editing

Routes: retain `/events/create`; edit from canonical event management/settings context.

Required wizard actions:

| Step | Required user actions | Relevant controls |
| --- | --- | --- |
| Basics | Title, generated slug, type, domain, description, cover, visibility | `Save draft`, `Continue`, `Cancel`, image upload/remove/replace. |
| Schedule | Timezone, event dates, registration period, location/online details | Date picker, time inputs, `Add session`, schedule validation Alert. |
| Registration | Capacity, waitlist policy, form, invitation policy | Switches/selects, `Preview registration`, `Add question`, `Select form`. |
| Operations | Staff, checklist, budget, communications defaults | Combobox, checkboxes, `Add task`, `Assign staff`. |
| Certificates/review | Template version, eligibility policy, review summary | `Preview certificate`, `Save draft`, `Submit for review` or `Publish` when authorized. |

Requirements:

- A draft saves intentionally; no accidental creation on next-step click.
- Each step has inline validation and summary of incomplete required fields before final submission.
- Long-running image/template operations show real progress/error state.
- Leaving with unsaved changes uses confirmation.
- Only status transition actions shown for current role/state are enabled.

### Event overview

Route: `/events/[slug]`.

The current details page is a starting point. Rebuild it as a useful landing page with role-aware actions.

Member view:

- Event information, schedule, location/map/online link when available, capacity/status, registration deadline, accessibility/contact details, sessions, related announcements.
- Contextual registration panel: `Register`, `Join waitlist`, `Complete registration`, `Cancel registration`, or `Open pass` exactly according to server state.
- `Add to calendar` generates a valid local download/link only after event schedule is valid.
- “Share event” action copies a canonical public URL with success feedback.

Organizer view adds:

- Status Badge with `Publish`, `Submit for review`, `Request changes`, `Cancel`, or `Mark completed` only when valid.
- Summary cards: registrations/capacity, checked in, waitlist, sessions, certificate eligible, campaign delivery, upcoming deadline.
- Prominent `Manage event` and contextual `Open scanner` for event-day operation.
- Quick action buttons: `Add session`, `Invite attendees`, `Send update`, `Import roster`, `Export roster`, `Issue certificates`—show at most the 2–3 most relevant visibly; others go to overflow menu/manage tabs.

### Event management workspace

Route: `/events/[slug]/manage` with redirects from separate management destinations.

Tabs:

| Tab | Actions/features |
| --- | --- |
| Overview | Planning checklist, ownership/staff, key metrics, lifecycle controls, latest activity, upcoming deadlines. |
| Registrations | Filterable roster, add/invite/import/export, row detail Sheet, manual status override with reason, waitlist promotion, resend pass. |
| Sessions | Create/edit/reorder sessions, session attendance count, start scanner for session, schedule conflict warnings. |
| Scanner | Full `AttendanceScanner`, mode selection QR/barcode/HID/manual, current result, offline sync queue, recent scans, scanner staff controls. |
| Communications | Event-scoped campaign list, compose update/invitation/reminder/cancellation, audience preview, delivery results. |
| Certificates | Eligibility count, template version, issue selected/all, progress/failure/delivery status, revoke/reissue. |
| Settings | Event edit, visibility, forms, integration options where real, duplicate/archive/cancel controls, audit timeline. |

Requirements:

- The management title/header always includes event name, status, date, and `View public page` action.
- Resource actions in each tab mutate through authorized server APIs and update relevant counts.
- Staff need not navigate to a global scanner just to scan one known event.
- The scanner tab must not leave camera active when navigating away.

### Registrations

Required roster controls:

- Search name/email/pass code.
- Filters: confirmed, waitlist, checked in, cancelled, no-show; source; session attendance; date.
- Columns: person, state, registration time, pass/attendance, form completeness, source, action menu.
- Actions: view detail, resend pass, check in manually (reason), move/promote waitlist where policy permits, cancel/refund policy when applicable, issue/review certificate only when authorized.
- Toolbar: `Invite`, `Add attendee`, `Import CSV`, `Export CSV`, `Send communication`; all scope-aware.
- Add attendee does not silently create a privileged club member; it follows the safe guest/invitation policy in the blueprint.

## Attendance scanner

Routes: canonical event management Scanner tab and `/events/[slug]/scanner` shortcut.

Required controls:

- Mode selector: `Camera`, `Hardware scanner`, `Manual entry`; QR/barcode formats are capabilities of Camera mode, not separate pages.
- Start/stop camera; switch camera if supported; upload image fallback; manual pass input; focused HID input field.
- Current event/session selector locked to authorized scope; session selection optional where event-level check-in is intended.
- Live result panel: person name/avatar only when authorized, outcome, time, status, clear success/error styling/text, optional vibration/audio toggle with accessible non-audio feedback.
- Recent scans list with outcome and undo/override only if a server-defined permitted workflow exists. Do not add a fake undo for immutable security history.
- Offline queue count, sync now, failed/conflict review, and connection status.

No scanner action may accept a raw client event identifier without server token/event/role validation.

## Certificates

### Member certificate wallet

Route: `/certificates`.

Required actions:

- Search/filter own certificates by event/date/status.
- Certificate card/list: event, template/issuer, issue date, status, verification code excerpt, `View`, `Download`, `Copy verification link`.
- Revoked certificate displays reason safe for recipient and replacement link if present; no download button if policy disallows it.
- Empty state: “Certificates appear here after an organizer issues them” plus `Browse events` link.

### Admin/lead certificate management

Routes: canonical `/admin/certificates` plus event Certificates tab.

Template actions:

- `Create template`, `Edit draft`, `Preview`, `Publish version`, `Duplicate`, `Archive`, `View usage`.
- Designer actions: upload/replace background, add/edit/remove field, undo where designer supports it, save draft, validate, preview with fixture data, publish immutable version.
- A published version cannot silently change. “Create new version” is the explicit action.

Issuance actions:

- Filter eligibility; select recipients; `Issue selected`, `Issue all eligible`, `Preview recipients`, `Test render` for organizer only.
- Progress and results show queued/rendering/issued/delivered/failed counts; failures link to detail/retry action.
- `Resend delivery`, `Reissue`, and `Revoke` are separate explicit actions, with reasons/confirmations where required.

## Communications

Routes: `/communications` and event Communications tab. Replace “Jobs” in user-facing navigation and labels.

### Campaign list

Required controls:

- Header primary `Compose campaign`; secondary `Manage templates`/`View delivery issues` as permissions allow.
- Data Table: campaign, category, event/context, audience count, channels, creator, scheduled/sent time, status, delivery summary, action menu.
- Filters: state, category, channel, event, date, creator, failures.
- Row actions: view, duplicate as draft, edit draft, schedule/send, pause/cancel scheduled, retry failures, export delivery report, view activity.
- Status display must visibly distinguish sent from partial delivery/failed.

### Compose campaign

Use full page or large responsive surface; campaign composition is consequential.

Steps/actions:

1. Select category/context/template.
2. Compose title, email subject/body, in-app content, CTA; preview each chosen channel.
3. Build audience (members/roles/domain/event state/selected rows) and load exact recipient preview/count/exclusions.
4. Send test to authorized recipient(s).
5. Choose send now/schedule; show timezone, recipient count, suppression policy, and final confirmation.

Required buttons: `Save draft`, `Preview`, `Send test`, `Back`, `Continue`, `Schedule`, `Send now`, `Cancel`. Buttons must reflect current validity/pending status.

## Members, recruitment, forms, resources

### Members

Route: `/admin/members` (or canonical role-aware equivalent).

Required actions:

- Search/filter members; table with identity, role/domain, verified/active state, event engagement summary, invite state.
- `Invite member`, `Add member`, `Edit role`, `Assign domain`, `View activity`, `Resend invite`, `Revoke session`, `Deactivate`/`Reactivate` when policy exists.
- Role/organization changes require confirmation and audit. Never place dangerous role change in an unconfirmed dropdown auto-save.
- Member detail Sheet shows profile, roles, membership history, assigned events/tasks, certificates, activity—not raw authentication secrets.

### Recruitment and forms

Required actions:

- Recruitment list: create cycle, open/close cycle, edit form, export applicants, review application, assign reviewer, schedule interview, accept/reject/request information, bulk email through Communications.
- Form builder: create/duplicate/archive form, add/reorder fields, preview as respondent, publish version, view responses, export authorized responses.
- Application review Sheet: applicant summary, form answers, reviewer notes/history, score only as assistive data, explicit decision/reason, next-step communication.

### Inventory, finance, projects, content

Each module gets a consistent list/overview/action pattern:

| Module | Primary action | Useful contextual actions |
| --- | --- | --- |
| Inventory | Add item | Adjust stock, assign, return, report loss, view log, request procurement, export. |
| Finance | Add expense/request | Attach receipt, submit, approve/reject with reason, record income, export ledger, view budget. |
| Projects | Propose project | Discuss, assign owner, change phase, add milestone/task, archive, link event. |
| Content | Create content item | Assign, schedule, attach asset, approve, publish reminder, import/export calendar. |

Do not introduce a control until its API/state/audit behavior is implemented. A clean list with five working actions is better than a dashboard of decorative cards.

## Notifications and activity

Transform the notification bell into an action-centered menu:

- Group unread/new/today/earlier.
- Each notification has type icon, title, short message, time, destination/action, and mark-read action.
- “Mark all read” asks no confirmation; it is reversible only in the sense of state, so show toast with optional undo if safely implemented.
- Critical system/delivery/import notifications link directly to a filtered resolution page.
- The full notifications page supports filter/read state and pagination.

## Cross-screen acceptance criteria

- Every primary action named in this document invokes a real, authorized backend operation or a real route to complete that operation.
- Every list has search/filter/empty/error/loading state appropriate to its data volume.
- Every high-impact action has explicit consequence text and confirmation.
- Every management action leaves a trace in the resource activity/audit timeline.
- No current placeholder, invalid route, or fake success control survives on an affected screen.
- The mobile layout preserves the primary action, statuses, and row/action access without horizontal overflow or inaccessible hover-only controls.
